## 1. 操作テストの方法と実測値

Playwright + Chromium（ヘッドレス）で iPhone SE 相当（375×667 / `isMobile` / `hasTouch`）にてゲームを起動し、HOME→RULE→READY→PLAYING→RESULT を実機操作に近い形でテストした。砂削りはマウスドラッグ、カメラは左バードラッグで再現。各画面のスクリーンショット取得、コンソール監視、フレーム時間サンプリングを実施。

| 観点 | 実測 | 評価 |
| --- | --- | --- |
| 起動 | DOMContentLoaded 74.5ms / load 74.7ms | 良好（単一HTML・外部依存なし） |
| 描画性能 | 平均 59.4fps（639サンプル, p95 17.1ms, ワースト98msは起動時の1回） | 良好 |
| 横スクロール | なし（scrollWidth=clientWidth=375） | 良好 |
| タップ範囲 | 主要ボタン 317×46px / リタイア 36px高 | おおむね良好（リタイアは小さめ） |
| 状態遷移 | HOME→READY→PLAYING→RESULT 正常、カウントダウン約2.6秒 | 良好 |
| 名前未入力ガード | 未入力時はHOMEに留まりエラー表示 | 良好 |
| ステージ検証 | `validateAllStages()` が **警告2件**（ステージ5・10） | **要修正** |

## 3. 各役割による課題抽出

### 検査担当
- **C-1（重大）**: `validateAllStages()` がステージ5「二段バー」・ステージ10「総合」で「ゴール直上を塞ぐ横長バーの可能性」を警告。`SPEC.md`「ゴール直上140px以内に横長バーを置かない」、`REVIEW_CHECKLIST.md`「validateAllStages() の警告が出ていない」に違反。原因は下段バー `{x:150, y:705, w:92}` がゴール（y=840）上端から135pxの位置で、ゴール幅の大半に重なるため。

### 操作性担当
- **O-1**: リタイアボタンが確認なしで即時終了（-5000点ペナルティ）。HUD上部の小さなボタンで、プレイ中の誤タップが致命傷になる。

### カラー/視認性担当
- **D-1**: 白っぽい🚬（fill `#fff8e5` / ring `#f8f0da`）が、掘った後に露出する空色グラデ背景（`#8bd6ff`→`#e4fff2`）や明るい砂の上で輪郭が溶けて見失いやすい。最も数の多い標準🚬の視認性が最弱。

### UX担当
- **U-1**: 残り時間に緊張感の演出がない。残り10秒以下でも色やサイズの変化がなく、時間切れが予告されない。
- **U-2（副次）**: HUDの残り時間が毎フレーム小数第1位まで更新され、最終桁が高速にちらつく。

### UI担当
- **I-1**: 「届けた」がHUDピル（`届けた 0/20`）と下部ステータス（`届けた 0 / 失った 0`）で重複。下部は「失った数」と注意喚起に振り分けた方が情報密度が上がる。

### 読み込み・堅牢性担当
- **L-1**: Supabase通信（`supa()`）にタイムアウトがない。回線が不安定な端末では「ランキング送信中…／読み込み中…」が無期限に残り続ける可能性。起動自体は74msで問題なし。

### 見送り（今回は対象外）
- スクロールバーの▲▼が押せそうに見えるが装飾（affordance）→ 影響軽微、別途検討。
- リタイアボタンの高さ36px→ 確認ダイアログ導入（O-1）で誤操作リスクが下がるため、サイズ変更は見送り。
- 自動操作では届けた数が伸びにくかったが、これは無作為な砂削りによるもので、戦略的に掘れば到達可能。難易度は仕様の範囲内と判断し変更しない。

## 4. 意見の擦り合わせと採用判断

各役割の指摘を「仕様適合性 × 効果 × リスク」で評価し、仕様/レビュー基準を壊さない範囲で以下を**採用**する。難易度バランスやステージ設計思想、ランキング仕様、絵文字や禁止表現には手を入れない。

| ID | 採用施策 | 担当役割 | 仕様影響 | リスク |
| --- | --- | --- | --- | --- |
| A | ステージ5・10の下段バーを `y:705→678` に上げ、ゴール直上140pxを開ける | 検査/実装 | 適合（警告解消） | 低 |
| B | 🚬全種に淡い暗色ドロップシャドウを付与し、背景を問わず輪郭を出す | カラー | 影響なし（色は不変） | 低 |
| C | リタイアに確認ダイアログを追加（誤タップ防止） | 操作性 | 「リタイアできる」維持 | 低 |
| D | 残り時間: 10秒超は整数、10秒以下は小数1位＋赤字で緊張感、ちらつき低減 | UX | 影響なし | 低 |
| E | 下部ステータスの「届けた」重複を解消し「失った数＋注意喚起」に | UI | 3要素以内維持 | 低 |
| F | `supa()` に8秒のタイムアウト（AbortController）を追加 | 読み込み/堅牢性 | 「1回だけ送信」「失敗時も壊れない」維持 | 低 |

## 5. 実装計画（index.html への具体修正）

### A. ステージ検証警告の解消
- `makeStage` 内、`if (variant === 5 || variant === 10) mechanisms.push({ ... y: 705 ... })` の `y` を `678` へ。
- 効果: ゴール上端(840)−140=700 の危険判定帯に下段バーが重ならなくなり、`validateAllStages()` の警告が解消。ゲート/ローターとの上下クリアランス（27px以上）も確保。二段バーの遊びは維持。

### B. 🚬の視認性（ドロップシャドウ）
- `drawCig()` の塗り前に、半透明の暗色円を下方向に少しずらして描画。
  - 例: `ctx.fillStyle='rgba(43,23,48,.26)'; ctx.beginPath(); ctx.arc(0,1.6,13,0,Math.PI*2); ctx.fill();`
- 効果: 空色背景・砂・トラップいずれの上でも各色🚬の輪郭が浮き、白系🚬を見失いにくい。色そのものは不変なので種別識別は保たれる。負荷は1円/個（最大30個）で無視可能。

### C. リタイア確認
- `retireBtn.onclick` を `if (state===PLAYING && confirm('リタイアすると-5000点のペナルティです。やめますか？')) finishGame('retire')` に。
- 効果: 誤タップでの-5000点を防止。確認中は `requestAnimationFrame` が止まり `dt` が上限0.033sで頭打ちのため、ダイアログ表示で時間が大きく減らない。

### D. 残り時間の表示
- `updateHud()` の時間表示を、`sec>10` は `Math.ceil(sec)`（整数）、`sec<=10` は `toFixed(1)`＋文字色 `#b53d35`。
- HTML初期値 `残り 60.0` を `残り 60` に揃える。
- 効果: 通常時は1秒刻みでちらつかず、終盤は0.1秒刻み＋赤字で緊張感。

### E. 下部ステータスの整理
- `statusHud` を `'失った ' + lost + '個　危険ゾーンと最下部に注意'` に。
- 効果: 「届けた」の重複を排し、上部HUDに無い「失った数」と注意喚起を提示。被失時の紫フラッシュ演出は維持。

### F. 通信タイムアウト
- `supa()` を `AbortController`＋`setTimeout(…,8000)` で包み、`finally` で `clearTimeout`。
- 効果: 回線不通時に8秒で打ち切り、既存のcatch経路（送信失敗/取得失敗の文言）へ確実に遷移。無期限スピナーを防止。「自動1回送信」「Publishable keyのみ」等は不変。

## 6. リスクと検証方法

- 回帰確認: 実装後に Playwright で再テストし、(1) `tabakonDebug.validateAllStages()` が空配列、(2) `pageerror`/`requestfailed` なし、(3) 平均FPSが従来同等、(4) HOME/PLAYING/RESULT のスクリーンショットで🚬視認性と残り時間表示を目視確認。
- 仕様回帰: `REVIEW_CHECKLIST.md` の該当項目（validateAllStages警告なし／リタイア可／色違い🚬／1回送信／通信失敗で壊れない／ステージ名表示）を満たすことを確認。
- 非対象: 難易度・ステージ設計・ランキングRPC・絵文字・禁止表現・Canvas表示サイズ(320×480)・2画面世界(320×960)は変更しない。
